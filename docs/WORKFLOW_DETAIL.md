# Note投稿くん — 実装フロー詳細

テーマを与えてから note.com 下書き投稿までの全フロー。

---

## 全体の流れ（所要時間：10〜15分）

```
あなた: 「テーマ：〇〇について記事を書いて」
  ↓
Phase 1: ディープリサーチ（3〜5分）
  ↓
Phase 2: 企画・構成設計（2分）
  ↓
Phase 3: 記事執筆（3〜5分）
  ↓
Phase 4: 画像生成（2〜3分）
  ↓
Phase 5: 品質チェック（1分）
  ↓
Phase 6: note.com 自動投稿（1〜2分）
  ↓
完成：下書きURL出力
```

---

## Phase 1 — ディープリサーチ

3エージェントを**並列**で起動する。

| エージェント | 役割 |
|-------------|------|
| `note-research` | note.com で同テーマの人気記事タイトル・構成を分析。スキが多い記事のパターンを抽出 |
| `gem-research` | Webから最新情報・統計・事例・データを9層で収集 |
| `mega-research-plus` | 8ソース統合で読者のペイン・悩みを調査。競合記事との差別化ポイントを特定 |

3つの結果を統合して「どの角度で書けばバズるか」を判断する。

---

## Phase 2 — 企画・構成設計

リサーチ結果をもとに以下を決定する。

### タイトル設計（3案 → 1つ選択）

| パターン | 例 |
|---------|---|
| 数字型 | 「〇〇する3つの方法」 |
| 問いかけ型 | 「なぜ〇〇は△△なのか？」 |
| 自己開示型 | 「正直に言うと、〇〇を知らなかった」 |
| 衝撃型 | 「〇〇した結果がやばすぎた」 |

ルール：30文字以内。「完全」「徹底」の乱用禁止。

### ハッシュタグ設計（5タグ）

| 優先 | ルール | 例 |
|------|--------|---|
| 1位（必須） | 主役の固有名詞そのまま | `Claude` |
| 2位（必須） | 関連固有名詞（会社名・バージョン等） | `Anthropic` |
| 3位 | note-researchで頻出したキーワード | `AI活用` |
| 4位 | ジャンル・ターゲット属性 | `エンジニア` |
| 5位 | 補完ワード | `生成AI` |

**禁止:** 主役でない競合製品名をタグにしない / 文末に `#タグ` 形式で書かない

### 記事構成

```
リード文（3〜5文）: 読者の悩みへの共感 or 体験の切り出し
  ↓
H2-1（問題提起）: 300〜500字 + 図解①
  ↓
H2-2（解決策）: 400〜600字 + 図解②
  ↓
H2-3（具体例）: 400〜600字
  ↓
H2-4（注意点）: 200〜400字 + 図解③
  ↓
まとめ: 200〜300字
```

合計目安：**1,500〜2,500字**

---

## Phase 3 — 記事執筆

`output/articles/記事名.md` として保存。

**フォーマット:**

```markdown
---
title: 記事タイトル（30字以内）
tags:
  - タグ1
  - タグ2
  - タグ3
  - タグ4
  - タグ5
---

リード文...

## 見出し1

本文...

![図解の説明](images/fig1.png)

## 見出し2

本文...
```

**AI臭消去ルール:**

- ✗ 「〜と言えるでしょう」「〜ではないでしょうか」禁止
- ✗ 「これにより」「このように」で段落を始めない
- ✗ 「です。」「ます。」を3回以上連続させない
- ✓ 語尾をばらす（です／だ／体言止め／口語／疑問形）
- ✓ 体験談・感情表現を必ず1箇所以上入れる
- ✓ 具体的な固有名詞・数値を入れる

---

## Phase 4 — 画像生成

`nanobanana2`（Gemini API: gemini-3.1-flash-image-preview）で生成。
`output/articles/images/` に保存。

| 種類 | 枚数 | 配置 |
|------|------|------|
| サムネイル | 1枚 | 記事のアイキャッチ |
| 図解（比較・フロー・まとめ） | 2〜3枚 | 各H2の直下 |

**プロンプトルール:**
- 日本語ゴシック体・白背景・パステルカラー・16:9横長
- 表示したいテキストをプロンプトに日本語でそのまま記載
- 1枚生成ごとに5〜10秒のインターバルを挟む

---

## Phase 5 — 品質チェック

投稿前に全項目を確認する。

- [ ] タイトルは30文字以内か
- [ ] リード文に「共感」「ベネフィット」が含まれるか
- [ ] AI臭い表現を使っていないか
- [ ] 体験ベースのエピソードが1つ以上あるか
- [ ] 図解が見出し直下に3枚以上入っているか
- [ ] 参考URLはWebFetchで200 OK確認済みか（不明なら削除）
- [ ] 文末に `#タグ` が混入していないか
- [ ] 文字数は1,500〜2,500字か

---

## Phase 6 — note.com 自動投稿

`publish-hybrid.js`（Playwright）が note.com を自動操作する。

```
1. 記事内画像を note.com にアップロード（API経由）
2. note.com エディターを自動で開く
3. サムネイルを設定
   └ 既存ドラフト更新時 → ×ボタンで旧サムネ削除 → 新規アップ
4. タイトルを入力
5. 本文を入力（図解画像を自動挿入）
6. 下書き保存
7. タグ5つを設定・保存
→ 完了：下書きURLを出力
```

**コマンド（新規）:**
```bash
"/c/Program Files/nodejs/node.exe" scripts/publish-hybrid.js \
  "output/articles/[article].md" \
  "output/articles/images/[slug]_thumb.png" \
  "draft" "output/articles" "[マガジン名]"
```

**コマンド（既存ドラフトのサムネのみ更新）:**
```bash
"/c/Program Files/nodejs/node.exe" scripts/publish-hybrid.js \
  "output/articles/[article].md" \
  "output/articles/images/[slug]_thumb.png" \
  "draft" "output/articles" "" "[ノートID]"
```

---

## 起動コマンド（最短）

```
「テーマ：[テーマ] で記事を下書きまで作ってください」
```

これだけで Phase 1〜6 が全自動で動く。
